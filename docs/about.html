Professional biography

The short story behind the profile

Serghei Fortuna is a Senior QA Automation Engineer and QA Lead whose work grew from practical testing into building systems that make product quality more visible, repeatable, and manageable.

Manual testing created the foundation: seeing the product through user behavior, noticing non-obvious scenarios, challenging requirements, and understanding where a real path can break. Automation added another layer: preserving team knowledge, reducing repetition, and stabilizing checks that should not depend on memory.

Over time, QA became an end-to-end model: how a team takes in requirements, removes uncertainty, chooses coverage, writes defects, trusts automation, sees release readiness, grows QA people, and uses AI without losing human accountability.

What shaped the approach

Agency work made systems necessary, not optional

The approach was shaped in an environment where several projects, teams, domains, stakeholders, and requirement maturity levels can run at the same time.

Parallel projects taught prioritization

When QA work spreads across multiple streams, personal attention alone does not scale. You need rules, templates, visible risks, shared language, and quick understanding of what matters most in each product.

Different domains taught adaptation

Enterprise data, healthcare migration, public-sector work, Drupal/CMS ecosystems, accessibility-heavy sites, and AI/RAG products do not need the same test strategy. The quality model must follow the risk.

Team growth taught communication

Hiring, role expectations, mentoring, live QA discussions, manual-to-automation growth, and shared standards became part of the work. Good QA leadership makes people stronger, not only processes stricter.

A recurring problem becomes a process the whole team uses, not a single knowledge holder.

Incomplete requirements → readiness rules. Unclear defects → a shared template. Noisy reports → normalized metrics. Brittle automation → architecture, data, owners. AI in QA → review gates and human ownership.

Working style

Calm, systematic, context-first

The work starts with practical questions: what is being built, for whom, where the risk is, what is already known, what must be clarified, who owns the decision, what can be automated, and what still requires human judgment.

Process is useful only when it reduces uncertainty. If a checklist is enough, use a checklist. If a manual task repeats, script it. If the team loses context, create a specification. If a report does not help decisions, change the report.

Communication Style

One quality goal, different stakeholder languages

The same risk reads differently for QA, engineering, product, and executives. Detail where it supports decisions; compression where it does not hide trade-offs.

With QA

Coverage, scenarios, data, defects, reproducibility, exploratory thinking, quality standards, and test design discipline.

With developers

Logs, environments, failure causes, stable locators, PR review, test architecture, CI behavior, and maintainability.

With PM / BA

Requirements, priorities, acceptance criteria, blockers, scope boundaries, assumptions, and readiness for development or UAT.

With leadership

Release readiness, accepted risks, QA capacity, hidden cost of chaos, project health, and evidence behind a go/no-go decision.
